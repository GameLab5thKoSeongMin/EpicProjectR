# Domain Model Proposal

## Design Principle

The new implementation should treat the game as an underwriting decision game first, and a Unity UI scene second. Domain logic should be pure C# where possible. ScriptableObjects should define authorable content, not mutable runtime state.

## Entities

- `ContractCase`: stable identity via `ContractId`.
- `Contract`: approved contract tracked over time.
- `TurnState`: current turn progress and reviewed slots.
- `PlayerCareerState`: money, evaluation score, rank, flags.
- `AfterStoryEventInstance`: queued event tied to source decision.

Entities have identity and can change over time.

## Value Objects

- `ContractId`, `RuleId`, `DocumentId`, `RouteId`, `TurnNumber`.
- `GameDate`.
- `MoneyAmount`.
- `DocumentFieldValue` with display text and stable code.
- `RuleActivationWindow`.
- `ReviewSubmission`.
- `ReviewResult`.
- `EconomyLineItem`.

Value objects should be immutable or readonly where practical.

## Services

- `UnderwritingReviewService`: evaluates a case against active rules.
- `RuleActivationService`: returns active rules for a turn, news, insurance type, and bundle.
- `DecisionScoringService`: compares player submission to review result.
- `PremiumPolicy`: applies CR-based premium multipliers.
- `OutcomeResolver`: deterministically resolves approved contracts using accident flags and return dates.
- `EconomyService`: calculates monthly evaluation and money changes.
- `AfterStoryScheduler`: emits channel events based on decisions/outcomes.
- `TurnProgressionService`: advances case slots, turns, news, and contract statuses.

Services should receive repositories and policies through constructor injection or composition root wiring.

## Policies / Rules

- `AbsoluteRejectionPolicy`: any active AR trigger requires rejection.
- `ConsiderationPricingPolicy`: CR count or CR IDs determine premium adjustment and recommended action.
- `CorrectActionPolicy`: maps review result to expected action.
- `AccidentPolicy`: approved plus `AccidentFlag = 1` causes accident.
- `NewsRulePolicy`: applies start/end turn and news overrides.
- `ScorePolicy`: objective criteria determine evaluation score; subjective deviations do not directly penalize.

## Repositories

- `ContractRepository`: contract case definitions by `ContractId`.
- `DocumentDefinitionRepository`: document bundles and fields.
- `RuleRepository`: AR/CR definitions.
- `NewsRepository`: scheduled news entries.
- `SpecialContractorRepository`: special profiles and arcs.
- `AfterStoryRepository`: after-story definitions.
- `LocalizationRepository`: optional if text is key-based.

For early milestones, repositories can wrap ScriptableObjects in memory. Later, importers can generate those definitions from Excel/CSV/JSON.

## Result Objects

- `ReviewResult`: triggered ARs, triggered CRs, active rules, suggested action.
- `SubmissionAuditResult`: checked correct, missed, false positives, action correctness.
- `PremiumQuoteResult`: base premium, multiplier, final premium, reasons.
- `OutcomeResult`: accident status, claim, return status, narrative triggers.
- `MonthlySettlementResult`: line items, totals, score delta.
- `ValidationResult`: errors, warnings, source references.

Use immutable/read-only collections in public APIs where practical.

## Runtime State

Runtime state should contain stable IDs and small snapshots, not ScriptableObject or prefab references:

- Current turn and slot.
- Reviewed contract decisions.
- Active contracts.
- Queued after-story events.
- Player career/economy.
- Unlocked news/codex entries.

## Content Definitions

ScriptableObjects are appropriate for:

- Rule definitions.
- Document definitions.
- Document bundle definitions.
- Contract case definitions if generated/imported.
- News definitions.
- Special contractor profiles.
- After-story event definitions.
- UI view configuration, separate from domain definitions.

ScriptableObjects should not be mutated as runtime state.

## Avoiding Over-engineering

Do not introduce ECS. The problem is data validation, deterministic rules, and UI flow, not high-volume entity simulation.

Do not default to global singletons. One composition root scene object can wire repositories, services, and presenters.

Do not create generic rule engines beyond the known needs. Start with explicit condition types: missing, equals, mismatch, numeric comparison, date comparison, route/news active, and cargo value ratio.
